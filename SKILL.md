# Скилл: Яндекс Wordstat

Сбор и **анализ** поискового спроса из Wordstat через `agent-browser`.
Навигация — через URL (`?words=<фраза>&region=<id>`), чтение — по snapshot.
Задача не в том, чтобы вывалить цифры, а чтобы отделить **целевой** спрос от шума.

## Философия (важно)

1. **Скепсис к нецелевому спросу.** Большие числа ≠ качественный трафик. Запрос
   может быть частотным, но приводить не тех людей.
2. **Думай как клиент.** Расширяй семантику творчески: синонимы, проблемы клиента,
   смежные формулировки.
3. **Всегда уточняй регион.** Спрос «ателье штор» в Москве и Минске несопоставим.
4. **Всегда проверяй интент.** См. раздел «Проверка интента».
5. **Показывай операторы** в отчёте — чтобы цифру можно было перепроверить.

## Использование

```
/yandex-wordstat <фразы> [--регион|--region <город>] [--тема <слаг>] [--out <csv>]
```

- **фразы** (обязательно) — одна или несколько базовых фраз. Разделитель `;` или перенос строки.
- **--регион / --region** — название региона. Дефолт: **Беларусь**. Маппинг в «Таблице регионов».
- **--тема** — слаг для имени папки результата. Дефолт: слаг из первой фразы.
- **--out** — путь к CSV. Дефолт: `<корень скилла>/research/<YYYY-MM-DD>-<тема>/keywords.csv`.

Пример: `/yandex-wordstat ателье штор; пошив штор на заказ --регион Минск`

## STOP — перед анализом

Если в команде **не задан регион** или **неясно, что продаёт клиент** — сначала спросить и дождаться ответа:

1. **Регион:** «Для какого региона смотрим спрос? Беларусь целиком / Минск / другой город?» (дефолт — Беларусь).
2. **Бизнес-цель:** «Что именно продаёшь/рекламируешь? Это нужно, чтобы отсеять нецелевые запросы.»

Без понимания бизнес-цели нельзя разделить целевые/нецелевые запросы — не пропускать этот шаг.

## Бинарь

`agent-browser` в `PATH` (проверка: `agent-browser --help`; нет — `npm i -g agent-browser`).
`node` нужен для URL-кодирования кириллицы.

## Таблица регионов (geo-id)

Подтверждённые id (проверены вживую):

| Регион | id |
|--------|-----|
| Беларусь (дефолт) | 149 |
| Минск | 157 |
| Москва | 213 |
| Россия | 225 |

Для региона не из таблицы — узнать id один раз вручную: открыть
`https://wordstat.yandex.ru` headed, сделать запрос, нажать кнопку региона,
выбрать нужный, **Подтвердить** — id появится в URL как `?region=<id>`. Записать сюда.
Не подставлять регион молча, если id неизвестен — спросить.

## Подготовка

1. Разобрать аргументы: фразы, регион (дефолт «Беларусь» → 149), тема, out.
2. Определить `region_id` по таблице.
3. Тема-слаг: из `--тема`, иначе из первой фразы (нижний регистр, пробелы → дефисы, кириллица сохраняется).
4. Дата: `date +%Y-%m-%d` (не выдумывать).
5. `mkdir -p "<корень скилла>/research/<дата>-<тема>/"`.

## Шаг 1. Проверка авторизации (один раз)

```bash
agent-browser --session yandex open "https://wordstat.yandex.ru/?words=%D1%82%D0%B5%D1%81%D1%82&region=149"
agent-browser --session yandex wait --load networkidle
agent-browser --session yandex snapshot -i
```

По snapshot:
- Ссылка «Выйти» + таблица с числами → залогинен, к Шагу 2.
- Форма «Log in with ID» / редирект на passport → НЕ залогинен.
- Капча → капча.

Если НЕ залогинен или капча:
1. Сообщить: «Wordstat просит войти/капчу. Открываю окно — залогинься руками».
2. `agent-browser --session yandex --headed open "https://passport.yandex.ru"`
3. Дождаться подтверждения от Анатоля.
4. Re-snapshot Wordstat, убедиться что данные доступны.

Бывает оверлей «Тур по интерфейсу» — закрыть («Close Tour» / крестик).
НИКОГДА не вводить пароль самостоятельно и не обходить капчу.

## Шаг 2. Сбор по каждой фразе

URL-кодировать фразу (кириллица!) через node:

```bash
enc=$(node -e 'process.stdout.write(encodeURIComponent(process.argv[1]))' "<фраза>")
```

### 2a. Базовая частотность + «Запросы со словами» (режим «Популярные»)

```bash
agent-browser --session yandex open "https://wordstat.yandex.ru/?words=$enc&region=<region_id>"
agent-browser --session yandex wait --load networkidle
agent-browser --session yandex snapshot -i
```

- **Первая строка** = сама фраза с числом → **базовая частотность** (тип `базовая`).
- **Все строки** таблицы = запросы, содержащие фразу → тип `со_словами`. Это
  **основной пул семантики** — прямые вариации твоего запроса.
- Длинная таблица — проскроллить (`scroll down`) и re-snapshot, добрать строки.

### 2b. Точная частотность (операторы)

Два уровня точности (см. «Операторы»):

```bash
# точная фраза (без лишних слов), словоформы свободны:
enc2=$(node -e 'process.stdout.write(encodeURIComponent(process.argv[1]))' "\"<фраза>\"")
# точная фраза + точные словоформы (предлоги фиксировать через !):
enc3=$(node -e 'process.stdout.write(encodeURIComponent(process.argv[1]))' "\"!<слово> !<слово>\"")
agent-browser --session yandex open "https://wordstat.yandex.ru/?words=$enc2&region=<region_id>"
agent-browser --session yandex wait --load networkidle
agent-browser --session yandex snapshot -i
```

Первая строка = **точная частотность** (тип `точная`). Это «чистый» спрос именно
на твою фразу — обычно намного меньше базовой.

### 2c. Похожие запросы (режим «Похожие»)

```bash
agent-browser --session yandex snapshot -i        # найти ref у label "Похожие"
agent-browser --session yandex click @<ref-Похожие>
agent-browser --session yandex wait --load networkidle
agent-browser --session yandex snapshot -i
```

Тип `похожая`. **Это семантически близкие запросы, НЕ обязательно содержащие твои
слова — часто шум** (алгоритм Яндекса уплывает в околотему). Полезно для
расширения, но интент обязательно проверять.

Между фразами пауза `agent-browser --session yandex wait 1500` (лимиты).
Пустой результат по фразе — пустая частотность в CSV, не падать.

## Шаг 3. Проверка интента (КРИТИЧНО)

**Прежде чем пометить запрос целевым — проверить, что человек хочет КУПИТЬ то, что
продаёт клиент, а не почитать/сделать самому/купить смежное.**

Для каждого перспективного запроса спросить себя:
1. Что человек хочет **купить** (не «чем интересуется»)?
2. Купит ли он **наш** продукт по этому запросу?
3. Или ищет что-то смежное/сопутствующее?

**Использовать WebSearch** для сомнительных запросов:
```
WebSearch: "<запрос>" что ищут покупатели
```
Смотреть: какие товары в выдаче? информационный или коммерческий интент?

### Красные флаги (скорее НЕ целевой)

- «**для** [продукта]» — нужен аксессуар/материал, не сам продукт.
- запрос про материалы/компоненты — человек делает сам.
- «**своими руками**», «**как сделать**», «как сшить» — информационный интент.
- «ремонт», «починить», «почистить» — уже владеет, сервисный запрос.

### Примеры

| Запрос | Выглядит как | На деле | Целевой? |
|--------|--------------|---------|----------|
| ателье штор | клиент ателье | клиент ателье | ✅ да |
| пошив штор на заказ | клиент ателье | клиент ателье | ✅ да |
| как сшить шторы своими руками | клиент ателье | DIY, шьёт сам | ❌ нет |
| ткань для штор купить | клиент ателье | покупает ткань, не услугу | ❌ нет |
| ремонт штор | клиент ателье | уже владеет, сервис | ⚠️ зависит |

(адаптировать таблицу под бизнес-цель, которую назвал клиент)

## Шаг 4. Записать результат

### keywords.csv

Колонки: `базовая_фраза,ключевая_фраза,тип,частотность,регион,интент`
- `тип` ∈ {`базовая`, `точная`, `со_словами`, `похожая`}
- `интент` ∈ {`целевой`, `нецелевой`, `?`} — по итогам Шага 3 (для проверенных; остальные `?`)
- UTF-8, разделитель — запятая, поля с запятыми — в двойных кавычках.
- Писать через инструмент Write.

### meta.md

```
# Wordstat выгрузка

- Дата: <дата>
- Регион: <регион> (geo-id <id>)
- Бизнес-цель: <что продаёт клиент>
- Исходные фразы: <список>
- Всего ключей: <число> (целевых: <n>, нецелевых: <n>)
```

## Шаг 5. Сводка в чат

**Разделить отчёт на целевые / нецелевые с обоснованием:**

```
✅ ЦЕЛЕВЫЕ (покупают то, что продаём):
- <фраза> — <частотность> показов
...

❌ НЕЦЕЛЕВЫЕ (ищут другое):
- <фраза> — <частотность> — <почему отсеяли>
...
```

Плюс по каждой базовой фразе: базовая и точная частотность. В конце — путь к CSV.

## Операторы Wordstat

| Оператор | Что делает | Пример |
|----------|-----------|--------|
| `"фраза"` | только эта фраза, без лишних слов | `"ателье штор"` |
| `!слово` | фиксирует словоформу | `!штор` (не «шторы») |
| `"!слово !слово"` | точная фраза + точные формы | `"!пошив !штор"` |
| `-слово` | исключить запросы со словом | `шторы -бесплатно -выкройка` |
| `(а\|б)` | объединить варианты | `(ателье\|пошив) штор` |

**Предлоги фиксировать через `!`:** `пошив !на заказ` (иначе «на» игнорируется).

### Стратегия анализа

1. Широко: `ателье штор` — общий объём.
2. Сузить кавычками: `"ателье штор"` — только фраза.
3. Зафиксировать формы: `"!ателье !штор"`.
4. Почистить минусом: `ателье штор -бесплатно -выкройка`.
5. Расширить: синонимы, проблемы клиента, смежные формулировки.

## Обработка ошибок

| Ситуация | Действие |
|----------|----------|
| Не залогинен / капча / протухшая сессия | Пауза, headed passport, просьба войти руками, продолжить |
| Пустой результат по фразе | Пустая частотность в CSV, идти дальше |
| Регион не в таблице | Узнать id вручную или спросить; не подставлять молча |
| `agent-browser`/`node` не найден | Подсказать `npm i -g agent-browser` |

## Чего НЕ делать

- Не хранить и не вводить пароли; вход — только руками.
- Не обходить капчу автоматически.
- Не помечать запрос целевым без проверки интента (Шаг 3).
- Не уходить в авто-расширение (похожие к похожим) — только то, что просили.
- Не коммитить `research/` — она в `.gitignore`.

## Заметки по UI (на момент 2026-06)

- Регион + запрос задаются через URL `?words=&region=`; ввод в поле и модал-дерево
  региона менее надёжны.
- Переключатель региона виден только на странице результатов (когда есть `words`).
